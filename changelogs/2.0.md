# LibCommand 2.0.0

Released June 16, 2026

**For PocketMine-MP API 5.0.0+ and PHP 8.2+**

2.0.0 is a complete rewrite of LibCommand, the advanced command framework for PocketMine-MP plugins. The entire library has been rebuilt from scratch with a new architecture focused on type safety, modularity, and separation of concerns.

It is **not** compatible with plugins written for version 1.x.X or lower. The old Commando-based system (`CortexPE/Commando`) has been entirely removed and replaced with a purpose-built command framework under the `imperazim\command` namespace.

## Why a Complete Rewrite?

The 1.x series was built on top of `CortexPE/Commando`, which imposed limitations on argument types, constraint systems, and dynamic command UI generation. Version 2.0 removes this dependency entirely, giving full control over:

- Command registration and lifecycle
- Argument parsing and validation with custom types
- Dynamic command UI via packet interception (using LibPacket)
- Constraint-based access control
- Subcommand nesting to arbitrary depth
- Error handling with structured failure objects

## Dependencies

LibCommand 2.0 depends on **LibPacket** for packet interception. It intercepts `AvailableCommandsPacket` to dynamically rebuild command overloads based on player permissions and constraints.

## Namespace Migration

All classes have moved from `imperazim\components\command` to `imperazim\command`.

| Old (1.x) | New (2.0) |
|---|---|
| `imperazim\components\command\Command` | `imperazim\command\Command` |
| `imperazim\components\command\SubCommand` | `imperazim\command\SubCommand` |
| Commando-based arguments | `imperazim\command\argument\*` |

## Core Architecture

### `imperazim\command\Command`

The abstract base class for all commands. Extends PocketMine's `pocketmine\command\Command` with three composable traits: `SubCommandTrait`, `ArgumentableTrait`, and `ConstraintableTrait`.

#### Highlights
- Added declarative `onBuild()` method that returns a configuration array instead of imperative setup.
- Added `onExecute(CommandResult $result)` for clean execution logic with parsed arguments.
- Added `onFailure(CommandFailure $failure)` for structured error handling with typed failure reasons.
- Added automatic `PermissionConstraint` and `CooldownConstraint` registration from config.
- Added `getMessage()` / `setMessage()` for custom failure messages.
- Added `getHelp()` with three format styles: `detailed`, `compact`, `usage`.
- Added automatic `CommandHistory::record()` on every execution.
- Removed dependency on `CortexPE/Commando`.

#### Changes
- **`onBuild()`**: Returns an associative array with keys: `name`, `description`, `aliases`, `permission`, `messages`, `constraints`, `cooldown`, `subcommands`, `arguments`. This replaces the old imperative `setup()` method.
- **`execute()`**: Fully handles subcommand routing, constraint checking, argument parsing/validation, history recording, and exception catching. Developers only implement `onExecute()`.
- **`getPlugin()`**: Returns the owning `Plugin` instance (stored as `readonly`).

#### Usage
```php
<?php

use imperazim\command\Command;
use imperazim\command\result\CommandResult;
use imperazim\command\result\CommandFailure;
use imperazim\command\argument\StringArgument;
use imperazim\command\argument\IntegerArgument;
use imperazim\command\constraint\InGameConstraint;

class GreetCommand extends Command {
    public function onBuild(): array {
        return [
            "name" => "greet",
            "description" => "Greet a player",
            "aliases" => ["hello", "hi"],
            "permission" => "myplugin.greet",
            "cooldown" => 5.0,
            "constraints" => [new InGameConstraint()],
            "arguments" => [
                new StringArgument("message", optional: false, description: "Greeting message"),
                new IntegerArgument("times", optional: true, default: 1, min: 1, max: 5),
            ],
        ];
    }

    public function onExecute(CommandResult $result): void {
        $sender = $result->getSender();
        $args = $result->getArgumentsList();
        $message = $args->get("message");
        $times = $args->get("times", 1);

        for ($i = 0; $i < $times; $i++) {
            $sender->sendMessage($message);
        }
    }

    public function onFailure(CommandFailure $failure): void {
        $failure->getSender()->sendMessage("Error: " . $failure->getMessage());
    }
}
```

### `imperazim\command\SubCommand`

Abstract class for subcommands. Supports nested subcommands (unlimited depth), independent constraints, and argument parsing.

#### Highlights
- Added `Command|SubCommand` union type for `$parent`, enabling nesting.
- Added `getPlugin()` that delegates to parent chain.
- Added `getSubCommands()` for child subcommand nesting.
- Added independent constraint and argument registration per subcommand.

#### Changes
- **`onBuild()`**: Returns config array with `name`, `description`, `aliases`, `permission`, `constraints`, `arguments`, `subcommands`.
- **`execute()`**: Handles child routing, constraint testing, argument validation, and delegates to `onExecute()`.
- **`onFailure()`**: Receives `CommandFailure` with `CONSTRAINT_FAILED` or `EXECUTION_ERROR` types.

#### Usage
```php
<?php

use imperazim\command\Command;
use imperazim\command\SubCommand;
use imperazim\command\result\CommandResult;
use imperazim\command\result\CommandFailure;
use imperazim\command\argument\PlayerArgument;

class TeleportSubCommand extends SubCommand {
    public function onBuild(): array {
        return [
            "name" => "tp",
            "description" => "Teleport to a player",
            "aliases" => ["teleport"],
            "permission" => "myplugin.admin.tp",
            "arguments" => [
                new PlayerArgument("target"),
            ],
        ];
    }

    public function onExecute(CommandResult $result): void {
        $sender = $result->getSender();
        $target = $result->getArgumentsList()->get("target");
        $sender->teleport($target->getPosition());
        $sender->sendMessage("Teleported to " . $target->getName());
    }

    public function onFailure(CommandFailure $failure): void {
        $failure->getSender()->sendMessage("Failed: " . $failure->getMessage());
    }
}
```

- Added `LibPacket::createInterceptor()` integration at `HIGHEST` priority.

### `imperazim\command\LibCommandInterceptor`

Intercepts `AvailableCommandsPacket` to dynamically modify the command UI shown to each player. Implements `PacketHandlerInterface` from LibPacket.

#### Highlights
- Added per-player constraint filtering: commands hidden from players who fail constraints.
- Added recursive overload generation for nested subcommands.
- Added argument combination (cartesian product) for multi-variant overloads.
- Added recursion guard via `$processedPackets` tracking.
- Added `CommandEnumManager` integration for soft enums in the command UI.

#### Changes
- **`handle()`**: Disassembles the packet, filters commands by constraints, rebuilds overloads, and re-sends the modified packet. Cancels the original.
- **`getOverloads()`**: Recursively generates `CommandOverload` objects from subcommand trees and argument combinations.

## Argument System

### `imperazim\command\argument\Argument`

Abstract base class for all argument types. Every argument has: name, optional flag, default value, description, aliases, custom validator, network parameter data, and tab completion support.

#### Highlights
- Added `$validator` callable for custom validation logic per argument.
- Added `getSuggestions()` for server-side tab completion (override in subclasses).
- Added `getUsageFormatted()` returning `<name: type>` or `[name: type]`.
- Added auto-generation of `CommandParameter` from `getNetworkType()` when not provided.
- Added `ArgumentException` thrown on name validation failure (no spaces allowed).

#### Changes
- **Constructor**: Named parameters: `$name`, `$optional`, `$default`, `$description`, `$aliases`, `$validator`, `$parameterData`.
- **`canParse()`**: Abstract method testing if a raw string is valid for this type.
- **`parse()`**: Abstract method converting raw string to typed value.
- **`runValidator()`**: Runs custom validator if set.

Accepts any string. Supports multi-token consumption (greedy) when it's the last positional argument with remaining tokens.

### `imperazim\command\argument\SoftEnumArgument`

Dynamic set of string choices that can be updated at runtime via `CommandEnumManager`. Unlike `EnumArgument`, values can be added/removed after registration.

#### Usage
```php
<?php

use imperazim\command\argument\SoftEnumArgument;
use imperazim\command\enum\CommandEnumManager;
use pocketmine\network\mcpe\protocol\types\command\CommandSoftEnum;

// Register the enum
CommandEnumManager::addEnum(new CommandSoftEnum("warps", ["spawn", "pvp", "shop"]));

// Use in argument
new SoftEnumArgument("warp", optional: false, enumName: "warps", description: "Warp name");

// Update later
CommandEnumManager::updateEnum("warps", ["spawn", "pvp", "shop", "arena"]);
```

### `imperazim\command\argument\PositionArgument`

## Constraint System

### `imperazim\command\constraint\Constraint`

Abstract base class for all constraints. Defines: `isSatisfiedBy()`, `onFailure()`, `onSuccess()`, `getDescription()`.

#### Highlights
- Added `onSuccess()` hook (default no-op) for tracking successful constraint checks (used by `CooldownConstraint`).
- Added `getDescription()` for auto-generated help text.

---

### `imperazim\command\constraint\PermissionConstraint`

Enforces permission node requirements. Automatically added to every command and subcommand.

#### Usage
```php
new PermissionConstraint("myplugin.admin", customMessage: "You lack admin access!")
```

---

### `imperazim\command\constraint\InGameConstraint`

Requires the sender to be an in-game `Player` (rejects console).

#### Usage
```php
new InGameConstraint(customMessage: "You must be in-game!")
```

---

### `imperazim\command\constraint\RequireConsoleConstraint`

Requires the sender to be the console (rejects players). Opposite of `InGameConstraint`.

#### Usage
```php
new RequireConsoleConstraint()
```

---

### `imperazim\command\constraint\CooldownConstraint`

Enforces a cooldown period between command uses per sender. Tracks usage via UUID for players and `'console'` for console. Supports custom messages with `{time}` placeholder.

#### Changes
- **`onSuccess()`**: Records the execution timestamp.
- **`isSatisfiedBy()`**: Cleans up expired entries, then checks remaining cooldown.

#### Usage
```php
new CooldownConstraint(
    cooldownSeconds: 5.0,
    customMessage: "Wait {time}s before using this again!"
)
```

---

### `imperazim\command\constraint\RateLimiterConstraint`

Limits command usage to N executions per sliding time window. Unlike `CooldownConstraint` (fixed delay), this allows burst usage up to the limit.

#### Usage
```php
// Max 3 uses per 60 seconds
new RateLimiterConstraint(maxUses: 3, windowSeconds: 60.0)
```

---

### `imperazim\command\constraint\GameModeConstraint`

Requires the player to be in a specific game mode (Survival, Creative, Adventure, Spectator).

#### Usage
```php
use pocketmine\player\GameMode;

new GameModeConstraint(GameMode::CREATIVE, customMessage: "Creative mode only!")
```

---

### `imperazim\command\constraint\WorldConstraint`

Restricts command usage to specific worlds. Accepts a single world name or array of names.

#### Usage
```php
new WorldConstraint(["lobby", "spawn"], customMessage: "Not available in this world!")
```

## Result System

### `imperazim\command\result\CommandResult`

Immutable value object passed to `onExecute()`. Contains: `$sender`, `$arguments` (as `CommandArguments`), `$label`.

#### Usage
```php
public function onExecute(CommandResult $result): void {
    $sender = $result->getSender();
    $args = $result->getArgumentsList();
    $label = $result->getLabel();
    $name = $args->get("name", "World");
    $sender->sendMessage("Hello, $name!");
}
```

---

### `imperazim\command\result\CommandFailure`

Structured failure object with typed reason constants: `INVALID_ARGUMENT` (0), `MISSING_ARGUMENT` (1), `CONSTRAINT_FAILED` (2), `EXECUTION_ERROR` (3), `COOLDOWN` (4).

#### Highlights
- Added `getReasonName()` for human-readable failure type.
- Added `getFailedConstraints()` for constraint failure introspection.
- Added `getMessage()` with fallback to type-specific defaults.
- Added `getData()` for full contextual error data.

#### Usage
```php
public function onFailure(CommandFailure $failure): void {
    $sender = $failure->getSender();
    match ($failure->getReason()) {
        CommandFailure::MISSING_ARGUMENT => $sender->sendMessage("Usage: " . $this->getUsage()),
        CommandFailure::INVALID_ARGUMENT => $sender->sendMessage("Bad argument: " . $failure->getMessage()),
        CommandFailure::EXECUTION_ERROR => $sender->sendMessage("Internal error occurred."),
        default => $sender->sendMessage($failure->getMessage()),
    };
}
```

## Dynamic Commands

### `imperazim\command\dynamic\DynamicCommand`

Builder class for creating commands without subclassing. Fluent interface with closures for `onExecute` and `onFailure`.

#### Highlights
- Added `DynamicCommand::create($plugin, $name)` factory method.
- Added fluent `withDescription()`, `withAliases()`, `withPermission()`.
- Added `addConstraint()`, `addArgument()`, `addSubCommand()` with method chaining.
- Added `setOnExecute(Closure)` and `setOnFailure(Closure)`.
- Added `registerCommand()` for one-step registration.

#### Usage
```php
<?php

use imperazim\command\dynamic\DynamicCommand;
use imperazim\command\argument\StringArgument;
use imperazim\command\constraint\InGameConstraint;
use imperazim\command\result\CommandResult;

DynamicCommand::create($plugin, "announce")
    ->withDescription("Make an announcement")
    ->withPermission("myplugin.announce")
    ->addConstraint(new InGameConstraint())
    ->addArgument(new StringArgument("message"))
    ->setOnExecute(function (CommandResult $result): void {
        $msg = $result->getArgumentsList()->get("message");
        $result->getSender()->getServer()->broadcastMessage($msg);
    })
    ->setOnFailure(function ($failure): void {
        $failure->getSender()->sendMessage("Error: " . $failure->getMessage());
    })
    ->registerCommand();
```

## Async Commands

### `imperazim\command\AsyncCommand`

Abstract base for commands that perform async operations (DB queries, HTTP requests). Dispatches work to PocketMine's `AsyncPool`, then calls back on the main thread.

#### Highlights
- Added `prepareAsyncData(array $args)` for serializable data preparation.
- Added `onAsyncComplete(CommandSender, mixed $data, array $args, string $label)` for main-thread completion.
- Added automatic player re-resolution (handles disconnect during async).

#### Usage
```php
<?php

use imperazim\command\AsyncCommand;
use pocketmine\command\CommandSender;
use imperazim\command\result\CommandFailure;

class LookupCommand extends AsyncCommand {
    public function onBuild(): array {
        return [
            "name" => "lookup",
            "description" => "Lookup player data",
            "arguments" => [new \imperazim\command\argument\StringArgument("name")],
        ];
    }

    protected function prepareAsyncData(array $args): mixed {
        return ["playerName" => $args["name"]];
    }

    public function onAsyncComplete(CommandSender $sender, mixed $data, array $args, string $label): void {
        $sender->sendMessage("Lookup complete for: " . $data["playerName"]);
    }

    public function onFailure(CommandFailure $failure): void {
        $failure->getSender()->sendMessage("Error: " . $failure->getMessage());
    }
}
```

## Command Scheduler

### `imperazim\command\scheduler\CommandScheduler`

Schedules command execution in the future using PocketMine's task scheduler.

#### Highlights
- Added `delay(sender, command, ticks, id)` for one-shot delayed execution.
- Added `repeat(sender, command, ticks, id, maxRuns)` for repeating execution with optional limit.
- Added `cancel(id)` and `cancelAll()`.
- Added `isActive(id)` and `getActiveTasks()`.

#### Usage
```php
<?php

use imperazim\command\scheduler\CommandScheduler;

CommandScheduler::init($plugin);

// Run a command after 5 seconds (100 ticks)
CommandScheduler::delay($sender, "broadcast Game starting!", 100);

// Repeat every second, max 10 times
CommandScheduler::repeat($sender, "say tick!", 20, "heartbeat", maxRuns: 10);

// Cancel
CommandScheduler::cancel("heartbeat");
```

## Command Wizard

### `imperazim\command\wizard\CommandWizard`

Interactive step-by-step command wizard via chat. Players answer questions one at a time with collected data passed to a completion handler.

#### Highlights
- Added `step(question, handler)` for adding wizard steps.
- Added `onComplete(Closure)` and `onCancel(Closure)` callbacks.
- Added `setData()` / `getData()` / `getAllData()` for wizard context.
- Added `cancel` keyword support for aborting.
- Added automatic cleanup on player disconnect.
- Added progress indicator (`1/3`, `2/3`, etc.).
- Added one-wizard-per-player guard (new wizard cancels existing).
- Implements `Listener` to capture `PlayerChatEvent`.

#### Usage
```php
<?php

use imperazim\command\wizard\CommandWizard;
use pocketmine\player\Player;

$wizard = new CommandWizard($plugin, $player, "Setup Warp");
$wizard->step("What is the warp name?", function (Player $p, string $answer, CommandWizard $w) {
    $w->setData("name", $answer);
    return true; // proceed to next step
});
$wizard->step("Is it public? (yes/no)", function (Player $p, string $answer, CommandWizard $w) {
    if (!in_array(strtolower($answer), ['yes', 'no'])) {
        $p->sendMessage("Please answer yes or no");
        return false; // repeat this step
    }
    $w->setData("public", strtolower($answer) === 'yes');
    return true;
});
$wizard->onComplete(function (Player $p, array $data) {
    $p->sendMessage("Warp '{$data['name']}' created! Public: " . ($data['public'] ? 'Yes' : 'No'));
});
$wizard->onCancel(function (Player $p) {
    $p->sendMessage("Wizard cancelled.");
});
$wizard->start();
```

## Command Macros

### `imperazim\command\macro\CommandMacro`

Allows players to create command aliases that execute multiple commands in sequence. Session-based (not persisted).

#### Highlights
- Added `create(player, name, commands)` with limits: max 20 macros, max 10 commands per macro.
- Added `execute(player, name)` dispatching all commands in sequence.
- Added `remove()`, `exists()`, `getList()`.
- Added `cleanup()` for player disconnect.
- Auto-normalizes leading `/` from commands.

#### Usage
```php
<?php

use imperazim\command\macro\CommandMacro;

// Create a macro
CommandMacro::create($player, "attack", ["/kit pvp", "/heal", "/tp arena"]);

// Execute it
CommandMacro::execute($player, "attack");

// List all macros
$macros = CommandMacro::getList($player);
```

## Traits

### `imperazim\command\traits\SubCommandTrait`

Provides subcommand management: `addSubCommand()`, `getSubCommands()`, `getSubCommand(name)`.

### `imperazim\command\traits\ArgumentableTrait`

Provides argument management: `addArgument()`, `getArguments()`, `getArgument(name)`. Handles raw argument parsing (greedy string, position token consumption), validation (required/excess/type), and packaging into `CommandArguments`.

### `imperazim\command\traits\ConstraintableTrait`

Provides constraint management: `addConstraint()`, `getConstraints()`, `getConstraintsByType()`, `testConstraints()`. Calls `onSuccess()` for all constraints when all pass, `onFailure()` for all failed constraints when any fail.

line. It replaces simple command wrappers with a full command composition
system: typed arguments, nested subcommands, execution constraints, command
results, dynamic builders, macros, scheduling and Bedrock client command UI
integration through LibPacket.

The goal is to make large command trees predictable. Each command or subcommand
can own its parsing, constraints, permissions and help text instead of forcing
plugins to keep everything inside one long `execute()` method.

### Added

- Added class-based `Command` and `SubCommand` foundations with build-time
  configuration, permissions, aliases, typed arguments and nested subcommands.
- Added typed arguments for strings, numbers, booleans, players, targets, items,
  worlds, positions, ranges, enums and soft enums.
- Added `CommandResult`, `CommandFailure` and `CommandSuccess` response objects
  so commands can return structured outcomes instead of only sending messages.
- Added `CommandResult::getRawArguments()` for commands that need the original
  trailing text, such as parse, broadcast, execute or scripting commands.
- Added constraints for permissions, cooldowns, rate limits, sender type, world
  checks and custom predicates.
- Added dynamic command builders for runtime-created command surfaces.
- Added command macros, command scheduling, command history and interactive
  wizard helpers.
- Added automatic help generation in detailed, compact and usage-only modes.
- Added permission tree utilities for inspecting command access.
- Added batch registration through `CommandGroup`.
- Added LibPacket-powered `AvailableCommandsPacket` interception for dynamic
  Bedrock autocomplete and command suggestions.
- Added regression tests for interceptor ownership, enum argument names and raw
  argument handling.

### Changed

- Changed the recommended command structure to "one command or subcommand per
  class". This keeps large APIs such as LibPlaceholder and LibWorld readable and
  makes each permission/action easier to test.
- Changed enum argument assembly to always use stable non-empty parameter names,
  avoiding malformed client command metadata.
- Changed the interceptor lifecycle so it is explicitly owned by the plugin that
  registered it.
- Changed the standalone plugin to require `LibPacket`, because client command
  UI integration depends on packet interception.
- Changed sync workflows to stage untracked files, rebase on remote changes and
  retry pushes when EasyLibrary receives concurrent embedded updates.

### Fixed

- Fixed command interceptor shutdown so standalone and embedded hosts can unload
  cleanly during reload or plugin disable.
- Fixed stale interceptor ownership when EasyLibrary switches between embedded
  and standalone command hosts.
- Fixed enum and soft enum parameter naming before packet assembly.
- Fixed quality analysis stubs and Composer lock state for reproducible CI.

### Compatibility Notes

- Requires PocketMine-MP API `5.0.0+`.
- Requires PHP `8.2+`.
- Standalone `LibCommand` depends on `LibPacket`.
- EasyLibrary embeds the same `imperazim\command` namespace and can provide the
  command API when the standalone plugin is not installed.

### Migration Notes

- Prefer one file per command and subcommand:

```txt
src/myplugin/command/MainCommand.php
src/myplugin/command/subcommand/ReloadSubCommand.php
src/myplugin/command/subcommand/DebugSubCommand.php
```

- Register command groups from your plugin owner:

```php
use imperazim\command\CommandGroup;

CommandGroup::register($this, [
    new MainCommand($this),
]);
```

### Example

```php
use imperazim\command\Command;
use imperazim\command\argument\StringArgument;
use imperazim\command\result\CommandFailure;
use imperazim\command\result\CommandResult;

final class GreetCommand extends Command {
    public function onBuild(): array {
        return [
            'name' => 'greet',
            'description' => 'Send a greeting',
            'permission' => 'example.greet',
            'arguments' => [
                new StringArgument('name'),
            ],
        ];
    }

    public function onExecute(CommandResult $result): void {
        $name = (string) $result->getArgumentsList()->get('name');
        $result->getSender()->sendMessage('Hello ' . $name . '!');
    }

    public function onFailure(CommandFailure $failure): void {
        $failure->getSender()->sendMessage('Command failed: ' . $failure->getReasonName());
    }
}
```
